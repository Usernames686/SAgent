## 1. 技术约束规范

### 1.1 语言统一约束（强制）

| 约束项 | 规范要求 | 验证方式 |
|--------|----------|----------|
| 后端服务语言 | 仅允许 Node.js (TypeScript)，禁止 Python/Java/Go 等 | CI 代码扫描，非 TS 文件拒绝合并 |
| 前端框架 | Next.js 15 (React + TypeScript) | package.json 依赖检查 |
| Agent 实现 | 所有 Agent 逻辑使用 TypeScript，通过 LLM Gateway 调用 AI 模型 | 代码审查 + CI 检查 |
| 数据处理 | 使用 Node.js 流处理/Worker Threads，禁止 Python 脚本 | CI 扫描 .py 文件 |
| 定时任务 | 使用 BullMQ (Node.js)，禁止 Celery/cron+Python | 部署配置审查 |
| 唯一例外 | 代码执行沙箱内部可运行用户提交的任意语言代码 | 沙箱隔离，不影响主服务 |

### 1.2 技术栈选型

| 技术领域 | 首选方案 | 备选方案 | 选型依据 |
|----------|----------|----------|----------|
| 前端框架 | Next.js 15 (App Router) + TypeScript | Nuxt 3 / Remix | SSR/SSG、React 生态 |
| UI 库 | Radix UI + Tailwind CSS 4 | shadcn/ui + Ant Design | 无障碍优先、高度可定制 |
| 状态管理 | Zustand + React Query | Jotai + SWR | 轻量、服务端/客户端分离 |
| 代码编辑器 | Monaco Editor | CodeMirror 6 | VS Code 同引擎 |
| 后端框架 | NestJS (Node.js) | FastAPI (Python) / Go Gin | TS 全栈统一、模块化 |
| 实时通信 | Socket.IO (Node.js) | native WebSocket | 自动重连、房间管理 |
| 消息队列 | BullMQ (基于 Redis) | RabbitMQ | 纯 Node.js 实现 |
| 代码执行 | Docker + gVisor | Firecracker | 安全隔离、多语言 |
| 向量检索 | Milvus Node.js SDK | pgvector | 专业向量检索 |
| 测试框架 | Vitest + Playwright | Jest + Cypress | 速度快、E2E 可靠 |

## 3. 智能体系统详细设计

### 3.1 智能体分层架构

| 层级 | 名称 | 包含 Agent | 核心职责 |
|------|------|-----------|----------|
| L0 | 元认知层 | Evolution Agent | 系统自我进化、策略优化、A/B 测试管理 |
| L1 | 编排调度层 | Orchestrator Agent | 意图识别、任务分解、Agent 调度、冲突解决 |
| L2 | 核心业务层 | Tutor、Evaluator、Path Planner、Debug、Code Review、Knowledge Graph、Interview、Mentor | 各专业化业务处理 |
| L3 | 基础能力层 | LLM Gateway、RAG Engine、Tool Executor、Memory Store | 基础 AI 能力支撑 |

### 3.2 Orchestrator Agent 设计

**职责**：全局编排调度，意图识别，Agent 路由，上下文管理，冲突解决

**意图识别决策表**：

| 意图类别 | 关键词/信号 | 路由 Agent | 协作模式 |
|----------|------------|-----------|----------|
| 概念提问 | "什么是..."、"怎么理解..."、"解释一下" | Tutor Agent | 独立 |
| 代码出错 | 运行报错、错误日志、异常栈 | Debug → Tutor | 串行 |
| 代码提交 | 提交评测、运行测试 | Evaluator + Code Review | 并行 |
| 路径请求 | "推荐学习"、"下一步学什么" | Path Planner + Knowledge Graph | 串行 |
| 面试准备 | "面试题"、"模拟面试" | Interview Agent | 独立 |
| 氛围描述 | 视觉风格、交互效果、Vibe 关键词 | Tutor Agent（氛围模式） | 独立 |
| Prompt 优化 | "帮我写 Prompt"、"优化提示词" | Tutor Agent（Prompt 模式） | 独立 |
| 闲聊/问候 | 非编程相关对话 | Tutor Agent（轻量模式） | 独立 |

**决策参数**：

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| confidence_threshold | float | 0.7 | 意图识别置信度阈值 |
| max_parallel_agents | int | 3 | 最大并行 Agent 数量 |
| agent_timeout_ms | int | 30000 | 单个 Agent 执行超时 |
| fallback_enabled | bool | true | 是否启用降级策略 |
| context_window_size | int | 8192 | 上下文窗口大小（Token） |

### 3.3 Tutor Agent 设计

**职责**：知识讲解与学习引导

**核心能力**：概念讲解、代码示范、苏格拉底式引导、多语言支持、解释层级适配

**进化路径**：
```
固定模板讲解 → 上下文适配讲解 → 个性化风格讲解 → 苏格拉底式引导 → 自主教学策略
```

**进化方向**：

| 维度 | 当前基线 | 3个月目标 | 6个月目标 | 12个月目标 |
|------|----------|-----------|-----------|------------|
| 辅导满意度 | 3.8/5 | 4.2/5 | 4.5/5 | 4.7/5 |
| 概念理解率 | 70% | 80% | 88% | 92% |
| 风格适配准确率 | 60% | 75% | 85% | 90% |
| 重复提问率 | 25% | 18% | 12% | 8% |

### 3.4 Evaluator Agent 设计

**职责**：代码质量评估

**核心能力**：正确性检测、风格评估、复杂度分析、测试用例生成、安全检测

**进化路径**：
```
规则匹配评估 → LLM 语义评估 → 多维度综合评估 → 专家级深度评估
```

**评估维度演进**：

| 阶段 | 评估维度 |
|------|----------|
| 基础 | 正确性 + 风格 |
| 进阶 | + 安全 + 性能 |
| 高级 | + 氛围匹配 + 可维护性 |
| 专家 | 全维度（8+） |

### 3.5 Path Planner Agent 设计

**职责**：学习路径规划与调整

**核心算法**：BKT（贝叶斯知识追踪）+ RL（强化学习）混合

**路径生成公式**：
```
推荐知识点 = argmax(
    P(掌握|推荐) × 学习收益权重
    - P(挫败|推荐) × 挫折惩罚权重
    + 路径连贯性奖励
)
subject to: 前置知识已掌握
```

**路径调整触发条件**：

| 触发条件 | 检测方式 | 调整动作 | 响应时间 |
|----------|----------|----------|----------|
| 连续 3 题失败 | 实时检测 | 回退到前置知识点巩固 | 即时 |
| 单题耗时 > 3 倍平均 | 实时检测 | 降低后续难度、提供提示 | 即时 |
| 连续 5 题通过 | 实时检测 | 加速推进、提升难度 | 即时 |
| 知识掌握率 < 60% | 阶段检测 | 路径重构、补充薄弱点 | 下一阶段 |
| 学习停滞 > 3 天 | 定时检测 | Mentor Agent 介入 | 24h 内 |
| 遗忘曲线触发 | 定时检测 | 推送复习任务 | 复习节点 |

### 3.6 Debug Agent 设计

**职责**：调试辅助与错误教学

**核心能力**：错误定位、原因分析、修复建议、调试教学、预防性提示

**进化路径**：
```
错误信息翻译 → 常见错误模式匹配 → 上下文感知诊断 → 预防性调试教学
```

### 3.7 Code Review Agent 设计

**职责**：代码审查与优化

**核心能力**：规范检查、最佳实践、重构建议、安全漏洞、性能优化

**进化路径**：
```
基础规范检查 → 代码异味检测 → 最佳实践推荐 → 架构级审查
```

### 3.8 Knowledge Graph Agent 设计

**职责**：知识图谱管理

**核心能力**：知识点关联、前置检测、缺口识别、推荐学习、图谱可视化

**知识图谱规模规划**：

| 语言/领域 | 知识点数量 | 关联边数量 | 覆盖等级 |
|-----------|-----------|-----------|----------|
| Python | 280+ | 450+ | L4（完整） |
| JavaScript/TypeScript | 250+ | 400+ | L4 |
| Java | 200+ | 320+ | L3 |
| C++ | 150+ | 240+ | L2 |
| Go | 120+ | 190+ | L2 |
| Rust | 100+ | 160+ | L1 |
| SQL | 80+ | 120+ | L3 |
| 数据结构与算法 | 180+ | 300+ | L4 |

### 3.9 Interview Agent 设计

**职责**：面试模拟与准备

**核心能力**：面试题生成、模拟面试、答案评估、薄弱点分析、简历建议

### 3.10 Mentor Agent 设计

**职责**：学习策略与动机管理

**核心能力**：策略建议、动机激励、时间管理、习惯养成、情绪识别

### 3.11 Evolution Agent 设计

**职责**：系统自我进化

**核心能力**：策略优化、效果评估、A/B 测试、参数调优、模式发现

**进化维度**：

| 进化维度 | 进化目标 | 进化方法 | 评估指标 | 进化周期 |
|----------|----------|----------|----------|----------|
| Prompt 策略 | 提升辅导质量 | Prompt 变体 A/B 测试 | 用户满意度、辅导有效性 | 每周 |
| 路由策略 | 提升意图识别准确率 | 决策树参数优化 | 意图识别准确率 | 每周 |
| 评估标准 | 提升评估准确度 | 评估结果与专家标注对比 | 评估准确率、误判率 | 每月 |
| 路径算法 | 提升学习路径效果 | 路径参数调优 + 多臂老虎机 | 路径完成率 | 每月 |
| 内容推荐 | 提升推荐精准度 | 协同过滤 + 内容特征 | 推荐点击率 | 每周 |
| 交互风格 | 提升用户体验 | 风格变体测试 | 交互满意度、留存率 | 每月 |

**进化安全约束**：
- 一键回滚：任何进化变更必须支持一键回滚到上一稳定版本
- 灰度发布：新策略必须经过灰度验证才能全量发布（1%→5%→20%→50%→100%）
- 效果下限：进化后的策略效果不得低于当前基线的 95%
- 人工审核：涉及 Prompt 核心逻辑变更需人工审核确认
- 审计日志：所有进化变更记录完整审计日志

### 3.12 Agent 通信协议（ACP/1.0）

```json
{
  "protocol": "ACP/1.0",
  "message": {
    "id": "msg-uuid",
    "from": "orchestrator",
    "to": "tutor",
    "type": "request|response|event|error",
    "timestamp": "2026-06-10T10:00:00.000Z",
    "correlationId": "session-uuid",
    "payload": {
      "action": "explain_concept",
      "parameters": { "concept": "闭包", "language": "javascript" },
      "context": {
        "userId": "user-uuid",

        "currentCode": "function counter() {...}",

      }
    },

      "priority": "high|medium|low",
      "timeout": 30000,
      "retryPolicy": { "maxRetries": 2, "backoffMs": 1000 }
    }
  }

```

**通信方式**：

| 通信方式 | 协议 | 适用场景 | 延迟要求 |
|----------|------|----------|----------|
| 同步 RPC | gRPC / HTTP | 需要即时响应的 Agent 调用 | ≤ 200ms |
| 异步消息 | RabbitMQ / Kafka | 不需要即时响应的任务 | 秒级 |
| 事件广播 | Event Bus | 状态变更通知 | ≤ 500ms |
| 共享状态 | Redis | 用户上下文共享 | ≤ 50ms |

### 3.13 Agent 协作模式

| 协作模式 | 描述 | 适用场景 | 示例 |
|----------|------|----------|------|
| 串行协作 | Agent 按序执行，前一个输出作为后一个输入 | 有明确依赖关系的任务 | Tutor 讲解 → Evaluator 评估 |
| 并行协作 | 多个 Agent 同时执行，结果由 Orchestrator 聚合 | 独立子任务 | Evaluator + Code Review 同时评估 |
| 条件协作 | 根据条件动态选择协作路径 | 需要判断的场景 | 代码提交 → 评估 → 若失败 → Debug |
| 递归协作 | Agent 可调用其他 Agent 完成子任务 | 复杂任务分解 | Path Planner 调用 Knowledge Graph |

### 3.14 冲突解决策略

| 冲突类型 | 解决策略 | 优先级规则 |
|----------|----------|------------|
| 评估结果冲突 | 加权投票，权重由 Agent 置信度决定 | 置信度高的 Agent 优先 |
| 建议矛盾 | Orchestrator 根据用户画像裁决 | 与用户学习风格匹配的优先 |
| 路径调整冲突 | 以 Path Planner 为主，其他 Agent 建议 | 路径一致性优先 |
| 资源竞争 | 请求队列 + 优先级调度 | 用户主动请求 > 系统主动推送 |

### 4.2 初始能力评估方案

| 评估阶段 | 评估内容 | 题目数量 | 时长 | 评估产出 |
|----------|----------|----------|------|----------|
| 阶段 1：逻辑思维 | 模式识别、逻辑推理、抽象思维 | 8 题 | 15 min | 逻辑思维分数 |
| 阶段 2：编程基础 | 按目标语言的基础语法、概念理解 | 10 题 | 20 min | 编程基础分数 |
| 阶段 3：问题解决 | 编码题（简单-中等难度） | 3 题 | 30 min | 编码能力分数 |
| 阶段 4：学习偏好 | 学习风格、节奏、目标问卷 | 15 项 | 5 min | 学习偏好画像 |

**自适应测试算法**：采用 Item Response Theory (IRT) 模型，根据用户答题情况动态调整后续题目难度，以最少题目精确评估能力水平。

### 4.5 路径生成算法

采用 **贝叶斯知识追踪 (BKT)** + **强化学习 (RL)** 混合算法：

1. **BKT 层**：基于用户答题数据，估算每个知识点的掌握概率 P(known)
2. **RL 层**：基于历史学习效果数据，优化知识点推荐序列
3. **约束层**：应用知识图谱的前置依赖约束

```
推荐知识点 = argmax(
    P(掌握|推荐) × 学习收益权重
    - P(挫败|推荐) × 挫折惩罚权重
    + 路径连贯性奖励
)
subject to: 前置知识已掌握
```

```
    │◀─────────────────────────── active
    │

    │
    │──用户放弃──▶ abandoned (已放弃)
    │
paused ──用户放弃──▶ abandoned
```

## 5. 氛围编程模块详细设计

### 5.1 Vibe Coding 范式概述

氛围编程（Vibe Coding）是一种以「说意图、定氛围、控结果」为核心的编程范式，用户通过自然语言描述意图和氛围，借助 AI 生成代码并掌控最终产出。

### 5.2 八大知识模块

| 模块 | 名称 | 难度范围 | 核心产出 |
|------|------|----------|----------|
| 1 | 认知与思维 | D1-D3 | 氛围描述文档（风格+功能+细节+技术约束） |
| 2 | 工具链 | D1-D3 | 独立搭建 AI 开发环境并部署应用 |
| 3 | 提示词工程 | D2-D4 | 高质量 Prompt 模板库 |
| 4 | 代码阅读理解 | D1-D3 | 能读懂、评审、修改 AI 生成代码 |
| 5 | 全栈工程能力 | D2-D3 | 独立交付完整全栈产品 |
| 6 | AI 高级能力 | D3-D4 | 构建 AI 增强应用 |
| 7 | 质量安全避坑 | D3-D4 | 保障代码质量与安全 |
| 8 | 实战项目 | D1-D4 | 可展示的项目作品集 |

### 5.3 交互模式设计

| 模式 | 描述 | 用户操作 | AI 行为 |
|------|------|----------|---------|
| 氛围描述式 | Vibe Coding 核心模式 | 输入氛围关键词+功能描述 | 生成匹配氛围的完整代码 |
| Prompt 迭代 | 多轮优化 Prompt | 描述→评审→优化 | 按优化后 Prompt 重新生成 |
| 配对编程 | 人机协作 | 人定方向、写 Prompt | AI 生成代码，人评审 |
| 氛围竞猜 | 反向训练氛围抽象 | 观察界面效果，反推描述 | 评分并给出优化建议 |
| 代码审查 | AI 代码评审训练 | 审查 AI 生成代码 | 提供评审清单和参考答案 |

### 5.4 实时反馈机制

| 反馈类型 | 触发条件 | 延迟要求 | 呈现方式 |
|----------|----------|----------|----------|

| 智能补全 | 用户暂停输入 | ≤ 300ms | 编辑器内联建议 |
| 氛围匹配度 | Prompt 提交后 | ≤ 2s | 氛围匹配评分 + 建议 |
| 幻觉预警 | AI 生成代码时 | ≤ 1s | 可疑代码高亮标注 |

## 13. 安全设计

### 13.1 代码沙箱安全架构

| 隔离层 | 技术 | 防护范围 |
|--------|------|----------|
| L1 容器隔离 | Docker + gVisor (runsc) | 系统调用过滤、内核隔离 |
| L2 资源限制 | cgroups + ulimit | CPU/内存/磁盘/网络限制 |
| L3 网络隔离 | NetworkPolicy + 无外网 | 禁止外联、仅内部通信 |
| L4 文件系统 | 只读根文件系统 + tmpfs | 禁止持久化写入 |
| L5 进程限制 | seccomp + AppArmor | 限制系统调用、禁止 fork bomb |

### 13.2 安全防护体系

| 安全域 | 威胁 | 防护措施 |
|--------|------|----------|
| 认证授权 | 未授权访问、Token 泄露 | JWT + Refresh Token、RBAC、OAuth 2.0、IP 白名单 |
| 数据传输 | 中间人攻击、窃听 | 全站 HTTPS/TLS 1.3、HSTS、CSP |
| 数据存储 | 数据泄露、拖库 | bcrypt 密码哈希、AES-256 加密敏感字段 |
| 代码执行 | 沙箱逃逸、恶意代码 | gVisor 多层隔离、资源限制、网络隔离 |
| AI 安全 | Prompt 注入、幻觉代码 | 输入过滤、输出审查、幻觉检测 |
| API 安全 | DDoS、暴力破解 | 限流（令牌桶）、WAF、验证码 |
| 隐私合规 | 隐私泄露、违规采集 | 数据脱敏、K-匿名化、差分隐私 |
