# sAgent 项目开发行为准则

> 减少大型语言模型（LLM）常见编码错误的行为准则。根据需要与特定项目的说明合并。
> 权衡：这些准则更倾向于谨慎而非速度。对于简单任务，请自行判断。

---

## 1. 编码前先思考

不要臆测。不要掩盖困惑。明确权衡取舍。

在实施之前：

- 明确陈述你的假设。如果不确定，就询问。
- 如果存在多种解释，将它们呈现出来——不要暗自选择。
- 如果存在更简单的方法，就说明。必要时要提出异议。
- 如果有不清楚的地方，就停下来。指出令人困惑的地方并询问。

## 2. 优先考虑简单性

用最少的代码解决问题。不做任何投机性的事情。

- 不添加超出要求的功能。
- 对于一次性使用的代码不进行处理抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能出现的场景进行错误处理。
- 如果你写了 200 行代码，而实际上 50 行就可以解决，那就重写。
- 问问自己："资深工程师会说这太复杂了吗？"如果答案是肯定的，那就简化。

## 3. 精准修改

只修改必须修改的内容。只清理自己造成的混乱。

编辑现有代码时：

- 不要"改进"相邻的代码、注释或格式。
- 不要对没有问题的部分进行重构。
- 即使你有不同的做法，也要遵循现有的风格。
- 如果你发现不相关的死代码，提及它——不要删除它。

当你的修改产生了孤立代码时：

- 移除因你的修改而不再使用的导入、变量和函数。
- 除非被要求，否则不要移除现有的死代码。

检验标准：**每一行修改都应直接与用户的请求相关。**

## 4. 目标驱动执行

定义成功标准。反复验证直至达标。

将任务转化为可验证的目标：

- "添加验证" → "为无效输入编写测试，然后让测试通过"
- "修复漏洞" → "编写一个能重现该漏洞的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都能通过"

对于多步骤任务，列出简要计划：

1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]

明确的成功标准能让你独立进行循环验证。模糊的标准（"让它工作"）则需要不断进行澄清。

---

## 验证准则是否生效

如果满足以下情况，则说明这些准则起作用了：

- 差异比较中不必要的修改更少
- 因过度复杂而进行的重写更少
- 澄清问题在实施之前提出而非在犯错之后
