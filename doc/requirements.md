## 2. 自我进化多智能体系统

### 2.1 系统概述

自我进化多智能体系统是平台的核心智能引擎，具备以下特征：
- **多智能体协作**：多个专业化 Agent 协同工作，覆盖学习全链路
- **自我进化**：系统基于用户反馈和学习效果数据，持续优化 Agent 策略与行为
- **自适应调整**：根据用户状态实时调整辅导策略和交互方式

### 2.2 智能体类型与职责边界

#### 2.2.1 智能体分层架构

```
┌─────────────────────────────────────────────────────────────────┐
│                     L0 — 元认知层 (Meta Layer)                   │
│  ┌─────────────────────────────────────────────────────────┐    │

│  │  职责：系统自我进化、策略优化、效果评估、A/B 测试管理       │    │
│  └─────────────────────────────────────────────────────────┘    │
├─────────────────────────────────────────────────────────────────┤
│                     L1 — 编排调度层 (Orchestration Layer)         │

├─────────────────────────────────────────────────────────────────┤
│                     L2 — 核心业务层 (Core Business Layer)        │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐          │

│  │ 智能辅导 │ │ 代码评估 │ │ 路径规划 │ │ 调试辅助 │          │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘          │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐          │
```

#### 2.2.2 智能体详细定义

| Agent | 层级 | 职责边界 | 核心能力 | 输入 | 输出 | 触发场景 |
|-------|------|----------|----------|------|------|----------|
| **Orchestrator** | L1 | 全局编排，不做业务逻辑 | 意图识别、任务路由、上下文传递、冲突解决、降级决策 | 用户消息 + 上下文 | Agent 调度指令 | 每次用户交互 |
| **Tutor** | L2 | 知识讲解与学习引导 | 概念讲解、代码示范、苏格拉底式引导、多语言支持、解释层级适配 | 问题 + 用户水平 + 当前代码 | 讲解内容 + 代码示例 | 用户提问、概念不理解 |
| **Evaluator** | L2 | 代码质量评估 | 正确性检测、风格评估、复杂度分析、测试用例生成、安全检测 | 代码 + 题目要求 | 评估报告 + 改进建议 | 用户提交代码 |
| **Path Planner** | L2 | 学习路径规划与调整 | 能力诊断、路径生成、动态调整、目标分解、路径推荐 | 用户画像 + 知识图谱 | 学习路径 + 调整建议 | 新用户注册、阶段评估 |
| **Debug** | L2 | 调试辅助与错误教学 | 错误定位、原因分析、修复建议、调试教学、预防性提示 | 错误代码 + 运行日志 | 诊断报告 + 修复方案 | 运行错误、逻辑错误 |
| **Code Review** | L2 | 代码审查与优化 | 规范检查、最佳实践、重构建议、安全漏洞、性能优化 | 代码 + 项目上下文 | 审查报告 + 优化建议 | 代码提交、项目评审 |
| **Knowledge Graph** | L2 | 知识图谱管理 | 知识点关联、前置检测、缺口识别、推荐学习、图谱可视化 | 查询 + 用户状态 | 知识关系 + 学习推荐 | 路径规划、概念查询 |
| **Interview** | L2 | 面试模拟与准备 | 面试题生成、模拟面试、答案评估、薄弱点分析、简历建议 | 目标岗位 + 用户水平 | 面试题 + 评估报告 | 面试准备模式 |
| **Mentor** | L2 | 学习策略与动机管理 | 策略建议、动机激励、时间管理、习惯养成、情绪识别 | 学习数据 + 行为模式 | 策略建议 + 激励内容 | 学习停滞、目标设定 |
| **Evolution** | L0 | 系统自我进化 | 策略优化、效果评估、A/B 测试、参数调优、模式发现 | 全局反馈数据 | 策略更新 + 参数调整 | 定期触发 + 阈值触发 |

### 2.3 智能体间协作机制

#### 2.3.1 协作模式

| 协作模式 | 描述 | 适用场景 | 示例 |
|----------|------|----------|------|
| **串行协作** | Agent 按序执行，前一个输出作为后一个输入 | 有明确依赖关系的任务 | Tutor 讲解 → Evaluator 评估 → Code Review 审查 |
| **并行协作** | 多个 Agent 同时执行，结果由 Orchestrator 聚合 | 独立子任务 | Evaluator + Code Review 同时评估代码 |
| **条件协作** | 根据条件动态选择协作路径 | 需要判断的场景 | 代码提交 → Evaluator 评估 → 若失败 → Debug 介入 |
| **递归协作** | Agent 可调用其他 Agent 完成子任务 | 复杂任务分解 | Path Planner 调用 Knowledge Graph 获取前置知识 |

#### 2.3.2 协作流程规范

```
用户请求
    │
    ▼
┌─────────────────┐
│ Orchestrator    │
│ 1. 意图识别     │
│ 2. 上下文加载   │
│ 3. 任务分解     │
│ 4. Agent 选择   │
└────────┬────────┘
         │
    ┌────▼────┐
    │ 路由决策 │
    └─┬───┬───┘
      │   │
  ┌───▼─┐ ┌▼────┐
  │串行 │ │并行 │
  │执行 │ │执行 │
  └───┬─┘ └┬────┘
      │    │
  ┌───▼────▼──┐
  │ 结果聚合   │
  │ 冲突解决   │
  │ 上下文更新 │
  └─────┬─────┘
        │
  ┌─────▼─────┐
  │ 统一输出   │
  │ 反馈收集   │
  └───────────┘
```

#### 2.3.3 冲突解决策略

| 冲突类型 | 解决策略 | 优先级规则 |
|----------|----------|------------|
| 评估结果冲突 | 加权投票，权重由 Agent 置信度决定 | 置信度高的 Agent 优先 |
| 建议矛盾 | Orchestrator 根据用户画像裁决 | 与用户学习风格匹配的优先 |
| 路径调整冲突 | 以 Path Planner 为主，其他 Agent 建议 | 路径一致性优先 |
| 资源竞争 | 请求队列 + 优先级调度 | 用户主动请求 > 系统主动推送 |

### 2.4 通信协议规范

#### 2.4.1 Agent 间通信协议

```json
{

    "payload": {
      "action": "explain_concept",

      "context": {

        "sessionId": "session-uuid",
        "userLevel": "beginner",

      }
    },
    "metadata": {
      "priority": "high|medium|low",
      "timeout": 30000,
      "retryPolicy": { "maxRetries": 2, "backoffMs": 1000 }
    }
  }

```

#### 2.4.2 通信方式

| 通信方式 | 协议 | 适用场景 | 延迟要求 |
|----------|------|----------|----------|
| **同步 RPC** | gRPC / HTTP | 需要即时响应的 Agent 调用 | ≤ 200ms |
| **异步消息** | RabbitMQ / Kafka | 不需要即时响应的任务 | 秒级 |
| **事件广播** | Event Bus | 状态变更通知 | ≤ 500ms |
| **共享状态** | Redis | 用户上下文共享 | ≤ 50ms |

#### 2.4.3 上下文共享规范

| 上下文类型 | 存储位置 | 生命周期 | 更新频率 |
|------------|----------|----------|----------|
| 用户会话上下文 | Redis | 会话期间 | 每次交互 |
| 用户长期画像 | PostgreSQL | 永久 | 每日批量 + 实时增量 |

### 3.5 路径动态调整算法

#### 3.5.1 路径生成算法

采用 **贝叶斯知识追踪 (BKT)** + **强化学习 (RL)** 混合算法：

1. **BKT 层**：基于用户答题数据，估算每个知识点的掌握概率 P(known)
2. **RL 层**：基于历史学习效果数据，优化知识点推荐序列
3. **约束层**：应用知识图谱的前置依赖约束

```
推荐知识点 = argmax(
    P(掌握|推荐) × 学习收益权重
    - P(挫败|推荐) × 挫折惩罚权重
    + 路径连贯性奖励
)
subject to: 前置知识已掌握
```

#### 3.5.2 路径调整触发条件

| 触发条件 | 检测方式 | 调整动作 | 响应时间 |
|----------|----------|----------|----------|
| 连续 3 题失败 | 实时检测 | 回退到前置知识点巩固 | 即时 |
| 单题耗时 > 3 倍平均 | 实时检测 | 降低后续难度、提供提示 | 即时 |
| 连续 5 题通过 | 实时检测 | 加速推进、提升难度 | 即时 |
| 知识掌握率 < 60%（阶段评估） | 阶段检测 | 路径重构、补充薄弱点 | 下一学习阶段 |
| 学习停滞 > 3 天 | 定时检测 | Mentor Agent 介入、调整策略 | 24h 内 |
| 用户主动请求调整 | 用户触发 | 重新评估、生成新路径 | 即时 |
| 遗忘曲线触发 | 定时检测 | 推送复习任务 | 复习节点 |

## 4. 氛围编程（Vibe Coding）学习内容

### 4.1 Vibe Coding 范式概述

氛围编程（Vibe Coding）是一种以「说意图、定氛围、控结果」为核心的编程范式，用户通过自然语言描述意图和氛围，借助 AI 生成代码并掌控最终产出。与传统编程不同，Vibe Coding 的核心能力在于**意图表达、氛围抽象、AI 协作与结果评审**，而非逐行手写代码。

**Vibe Coding 知识模块全景**：

```
认知思维 → 工具链 → 提示词工程 → 极简代码阅读 → 全栈工程能力 → AI 高级应用 → 质量安全 → 实战项目
```

### 4.2 八大知识模块体系

#### 4.2.1 模块一：认知与思维（最核心，决定上限）

| 知识点 | 内容 | 学习目标 | 难度 |
|--------|------|----------|------|

| **氛围（Vibe）抽象能力** | 把模糊感觉 → 具象关键词：风格、色调、质感、交互动效 | 能将抽象感受转化为 AI 可理解的描述 | D2 |

### 4.7 氛围编程实践设计规范

| 模式 | 描述 | 适用场景 | 交互特征 |

| **Bug 修复** | 给定有 Bug 的代码 | 调试训练 | 含 Bug 代码、错误现象、修复验证 |
| **代码审查** | 审查 AI 生成代码并提出改进 | 代码评审能力 | 待审查代码、评审清单、幻觉排查 |
| **氛围竞猜** | 给定界面效果，反推氛围描述 | 氛围抽象训练 | 视觉/交互效果、关键词匹配 |

#### 4.7.2 实时反馈机制

| 反馈类型 | 触发条件 | 延迟要求 | 呈现方式 |
|----------|----------|----------|----------|
| **语法错误** | 代码编辑时 | ≤ 500ms | 编辑器内联标注 |
| **Lint 提示** | 代码编辑时 | ≤ 1s | 编辑器波浪线 + 悬浮提示 |

| **AI 分析** | 代码提交后 | ≤ 3s (TTFT) | AI 面板流式输出 |
| **智能补全** | 用户暂停输入 | ≤ 300ms | 编辑器内联建议 |
| **氛围匹配度** | Prompt 提交后 | ≤ 2s | 氛围匹配评分 + 建议 |
| **幻觉预警** | AI 生成代码时 | ≤ 1s | 可疑代码高亮标注 |

### 4.9 内容更新机制

#### 4.9.1 内容生命周期

```
创建 → 评审 → 发布 → 监控 → 优化 → 归档
 │       │       │       │       │       │
 │       │       │       │       │       └─ 过时内容归档
 │       │       │       │       └─ 基于数据优化
 │       │       │       └─ 跟踪使用数据
 │       │       └─ 内容上线
 │       └─ AI + 人工审核
 └─ 作者创建 / AI 生成
```

#### 4.9.2 内容质量审核流程

| 审核阶段 | 审核内容 | 审核方式 | 通过标准 |
|----------|----------|----------|----------|
| **技术准确性** | 代码正确性、Prompt 有效性、概念准确性 | 自动测试 + 专家审核 | 100% Prompt 可复现、代码可运行 |
| **教学有效性** | 难度适当、循序渐进 | 教学专家审核 + 用户测试 | 80% 目标用户能完成 |
| **内容完整性** | 描述清晰、示例充分 | AI 检查 + 人工审核 | 无缺失信息、示例覆盖关键场景 |
| **安全合规** | 无安全隐患、无不当内容 | 自动扫描 + 人工审核 | 通过安全扫描、无违规内容 |
| **时效性** | 工具版本、API 变更、最佳实践 | 季度审查 + 社区反馈 | 内容与当前工具版本一致 |

#### 4.9.3 版本控制策略

- **语义化版本**：Major.Minor.Patch（如 2.1.3）
  - Major：知识模块体系重大变更
  - Minor：新增内容或较大修改
  - Patch：错误修正、小优化
- **版本兼容**：保留最近 2 个 Major 版本
- **迁移支持**：提供版本间学习进度迁移方案
- **变更通知**：内容更新后通知受影响用户
- **工具版本追踪**：AI IDE / 云平台等工具更新后及时同步内容

| **连接池** | 数据库连接池（HikariCP）、HTTP 连接池 | 后端服务 |
| **缓存策略** | Redis L1 + 本地缓存 L2，TTL + LRU 淘汰 | 热点数据 |
| **异步处理** | 消息队列解耦，非核心路径异步化 | AI 评估、行为采集 |
| **限流降级** | 令牌桶限流 + Sentinel 降级 | 流量突增 |
| **弹性扩缩** | K8s HPA 基于 CPU/内存/自定义指标自动扩缩 | 负载波动 |
| **读写分离** | PostgreSQL 主从 + 读写路由 | 数据库压力 |
| **CDN 加速** | 静态资源 CDN + 边缘缓存 | 前端资源 |

### 6.6 容灾备份策略

| 策略 | 实施方案 | RPO | RTO |
|------|----------|-----|-----|
| **数据备份** | 每日增量 + 每周全量，异地存储 | ≤ 24h | ≤ 4h |
| **多可用区** | 同区域双 AZ 部署 | 0 | ≤ 30s |
| **故障转移** | 自动健康检查 + 故障转移 | 0 | ≤ 60s |
| **灾备演练** | 季度灾备演练 | - | - |

### 6.7 水平扩展机制

| 组件 | 扩展方式 | 扩展单元 | 扩展触发条件 |
|------|----------|----------|-------------|

## 9. 典型用例与流程图

### 9.1 用例 UC-001：新用户注册与路径生成

| 项目 | 内容 |
|------|------|
| **用例编号** | UC-001 |
| **用例名称** | 新用户注册与路径生成 |
| **角色** | 新用户 |
| **前置条件** | 用户未注册，访问平台首页 |
| **后置条件** | 用户账户已创建，个性化学习路径已生成 |

**基本流程**：
1. 用户点击「注册」按钮
2. 系统展示注册表单（邮箱/手机 + 密码）
3. 用户填写并提交注册信息
4. 系统验证信息有效性，创建账户
5. 系统展示学习目标选择页
6. 用户选择学习目标（如：Python 零基础 → 数据分析师）
7. 系统展示能力诊断测试
8. 用户完成诊断测试（逻辑思维 + 编程基础 + 问题解决）
9. 系统基于测试结果和目标生成个性化学习路径
10. 系统展示学习路径概览
11. 用户确认路径，进入学习仪表盘

**扩展流程**：
- 3a. 信息无效 → 系统提示错误，用户修正后重新提交
- 5a. 用户选择「跳过」→ 系统生成默认路径，后续可调整
- 8a. 用户中途退出测试 → 系统保存进度，下次继续
- 9a. 测试结果不足以生成路径 → 系统补充推荐测试题

**异常流程**：
- 4a. 注册服务不可用 → 系统提示稍后重试
- 9a. 路径生成超时 → 系统展示默认路径，后台异步生成个性化路径

9. 系统展示评测结果（通过/失败 + 详情）
10. Evaluator Agent 展示 AI 评估报告
11. 用户查看改进建议或继续下一题

**扩展流程**：
- 4a. 用户请求 AI 提示 → Tutor Agent 提供渐进式提示
- 6a. 运行出错 → Debug Agent 展示错误分析和修复建议
- 9a. 部分测试未通过 → 系统展示失败用例详情
- 10a. 用户对评估有疑问 → 可请求详细解释

**异常流程**：
- 5a. 沙箱不可用 → 系统提示稍后重试
- 7a. 提交超时 → 系统提示重试，代码自动保存

### 9.5 用例 UC-005：系统自我进化

| 项目 | 内容 |
|------|------|
| **用例编号** | UC-005 |
| **用例名称** | 系统自我进化 |
| **角色** | Evolution Agent（自动触发） |
| **前置条件** | 系统运行 ≥ 2 周，有足够反馈数据 |
| **后置条件** | 策略已优化或回滚 |

**基本流程**：
1. Evolution Agent 定期分析全局反馈数据
2. 识别可优化维度（如 Tutor Agent 的 Prompt 策略）
3. 生成策略变体
4. 创建 A/B 测试实验（1% 流量）
5. 运行实验，收集数据
6. 统计分析实验结果
7. 若效果显著提升 → 渐进放量（5% → 20% → 50% → 100%）
8. 若效果无提升或下降 → 回滚变体
9. 记录进化日志

**扩展流程**：
- 6a. 结果不显著 → 延长实验周期
- 7a. 放量过程中效果下降 → 立即回滚
