## 1. 现状诊断与问题分析

### 1.1 当前知识点优先级分布

当前 `knowledge-seed.data.ts` 中 69 个知识点的优先级分布严重失衡：

| 优先级 | 定义 | 当前数量 | 占比 | 问题 |
|--------|------|----------|------|------|
| P0 | 核心必学 | ~47 | 68% | 数量过多，几乎所有 JS 基础和 React 基础都标为 P0，导致 P0 含金量下降 |
| P1 | 常用进阶 | ~13 | 19% | **严重不足**，缺少大量行业必备技能 |
| P2 | 进阶优化 | ~7 | 10% | 偏少，但可接受 |
| P3 | 底层原理 & 高级架构 | ~2 | 3% | 偏少，但可接受 |

### 1.2 P1 知识点严重缺失的具体领域

| 领域 | 当前 P1 覆盖 | 行业实际需求 | 差距 |
|------|-------------|-------------|------|
| JavaScript 进阶 | 仅 JS-013 错误处理 | 正则表达式、Map/Set、Proxy/Reflect、WeakMap/WeakSet、Symbol 深入、装饰器提案 | 缺 6+ 个核心知识点 |
| Node.js 服务端 | 仅 NODE-005 NestJS管道、NODE-009 WebSocket、NODE-017 日志 | 日志体系(Log4js/Winston)、文件上传、定时任务、环境变量管理、进程管理实践 | 缺 5+ 个核心知识点 |
| React 生态 | 仅 REACT-007 自定义Hook、REACT-009 TS+React、REACT-010 表单校验 | React Server Components、Next.js App Router、Zustand 状态管理深入、React Query/SWR 数据获取、国际化 i18n | 缺 5+ 个核心知识点 |
| 前端工程 | 仅 ENG-002 TS进阶、ENG-003 RESTful | 包管理器深入(pnpm/monorepo)、CI/CD 流水线、性能监控与优化、微前端基础 | 缺 4+ 个核心知识点 |
| Vibe Coding 核心 | **完全空白** | Prompt Engineering 深入、AI 代码评审、氛围描述方法论、AI 辅助调试 | 缺 8+ 个核心知识点 |
| 数据库 | 无 P1 | MongoDB 基础、数据库设计范式、SQL 查询优化 | 缺 3+ 个核心知识点 |

**P1 总缺口：至少 31 个知识点需要新增或从 P0/P2 调整**

### 1.3 学习方式的三大核心问题

#### 问题一：学习模式单一，缺乏沉浸感

| 维度 | 当前状态 | 应有状态 |
|------|---------|---------|
| 学习入口 | 知识点列表 → 直接看文本 | 分阶段引导 → 渐进式解锁 → 沉浸式学习环境 |
| 概念理解 | 纯文本罗列，无层次 | 动机引入 → 概念卡片 → 交互式代码示例 → 即时练习 |
| 动手实践 | 通用模板 + 正则检查（覆盖面窄、准确度低） | 场景化任务 + AI 评审 + 单元测试验证 + 逐步提示 |
| 评估反馈 | 2-3 道选择题（覆盖面不足） | 多题型（选择+填空+代码补全+代码评审）+ IRT 自适应出题 |
| 知识巩固 | 无 | 间隔重复提醒、错题回顾、关联知识点推荐 |

#### 问题二：练习系统粗糙

当前 `exercise-data.ts` 存在的问题：

| 问题 | 详情 | 影响 |
|------|------|------|
| 检查方式简陋 | 用正则表达式匹配代码模式，如 `/const\s+\w+\s*=/` | 只能检查"写了什么"，不能检查"写得对不对" |
| 无运行验证 | 练习代码不能实际运行 | 学生无法看到运行结果，无法验证逻辑正确性 |
| 无逐步提示 | 做不出来只能看参考答案 | 缺少脚手架式引导，学习效果差 |
| 无场景化 | 练习脱离真实开发场景 | 学生无法建立"什么时候用"的心智模型 |
| P1 知识点练习缺失 | 新增 P1 知识点没有对应练习 | 学习闭环断裂 |

#### 问题三：缺少 Vibe Coding 特色学习方式

需求文档定义了 9 种氛围编程交互模式（氛围描述式、引导式编码、配对编程、Prompt 迭代、代码补全、代码重构、Bug 修复、代码审查、氛围竞猜），但**当前系统一种都没有实现**。

### 1.4 知识体系结构问题

| 问题 | 现状 | 改进方向 |
|------|------|---------|
| P0 过多 | 47/69=68%，几乎所有 JS 基础都标 P0 | 重新划分，P0 应控制在 30-40% |
| 缺少 Vibe Coding 专属模块 | AI 模块仅 3 个知识点（AI-001~003） | 独立为完整模块，至少 15 个知识点 |
| 缺少数据库模块 | 数据库内容散落在 NODE-006 | 独立模块，系统覆盖 SQL + NoSQL |
| 缺少 DevOps 模块 | 仅 ENG-001 ESLint | 独立模块覆盖 CI/CD + Docker + 监控 |
| 知识点关联不足 | 仅有前置/后继关系 | 增加关联关系、对比关系、应用关系 |

## 2. 改进目标与设计原则

### 2.1 核心目标

| 目标 | 量化指标 | 当前 | 目标值 |
|------|---------|------|--------|
| P1 知识点数量 | 个 | ~13 | 40+ |
| 知识点总数 | 个 | 69 | 110+ |
| 学习模式 | 种 | 2（阅读+选择） | 7+ |
| 练习验证方式 | 种 | 1（正则匹配） | 4（正则+运行+AI评审+单元测试） |
| Quiz 题型 | 种 | 1（选择题） | 5（选择+填空+代码补全+代码评审+排序） |
| Vibe Coding 专属知识点 | 个 | 3 | 15+ |

### 2.2 设计原则

1. **渐进优先**：P0→P1→P2→P3 严格递进，P0 只放"不懂就不能继续"的知识
2. **实战导向**：每个 P1 知识点都有真实开发场景的练习
3. **闭环学习**：概念→示例→练习→评估→巩固，缺一不可
4. **氛围特色**：Vibe Coding 的学习方式本身就要体现"说意图、定氛围、控结果"
5. **可验证**：练习和评估必须能客观验证学习效果，而非仅凭正则匹配

## 4. 学习方式重构方案

### 4.1 从"2种模式"到"7种学习模式"

| 模式 | 当前状态 | 改进方案 | 适用知识点 |
|------|---------|---------|-----------|
| 📖 概念理解 | ✅ 已有(lecture) | 增加交互式概念卡片、动画演示、对比表格 | 全部 |
| 💻 动手实践 | ⚠️ 有但粗糙 | 场景化任务 + 4级验证 + 逐步提示 + AI评审 | 全部 |
| 📝 Quiz评估 | ⚠️ 仅选择题 | 5种题型 + IRT自适应出题 + 错题回顾 | 全部 |
| 🤖 AI对话辅导 | ✅ 已有 | 增加知识点上下文注入，AI知晓当前学习进度 | 全部 |
| 🎨 氛围编程模式 | ❌ 缺失 | 说意图→AI生成→评审→迭代，Vibe Coding核心 | VIBE-* |
| 🐛 Bug修复挑战 | ❌ 缺失 | 给含Bug代码，限时修复，排位榜 | NODE-*/REACT-* |

### 4.2 练习系统 4 级验证体系

#### Level 1：正则模式匹配（现有，保留但降级）

```typescript
// 当前方式，快速但粗糙
{ pattern: /const\s+\w+\s*=/, name: 'const 声明', weight: 0.2 }
```

- 用途：快速检查代码是否包含关键语法元素
- 适用：D1-D2 难度的入门知识点
- 权重：占评分的 20%

#### Level 2：代码运行验证（新增）

```typescript
interface RuntimeCheck {
  type: 'runtime';
  // 在沙箱中执行学生代码，对比输出
  testCases: Array<{
    input?: string;          // 标准输入
    expectedOutput: string;  // 期望输出
    description: string;
  }>;
  timeout: number;           // 超时时间(ms)
}
```

- 用途：运行代码并验证输出正确性

### 4.5 Vibe Coding 特色学习模式

#### 模式 A：氛围描述式（Vibe-Driven Development）

```
用户流程：
1. 选择目标（如"做一个登录页"）
2. 描述氛围（"极简、深色、毛玻璃、呼吸动效"）
3. AI 生成代码
4. 用户评审→调整氛围→再生成
5. 最终输出：氛围 Prompt + 生成代码 + 评审报告

学习映射：
- VIBE-004 氛围抽象能力
```

#### 模式 B：AI 配对编程（Pair Programming with AI）

```
用户流程：
1. 用户描述需求（用自然语言）
2. AI 生成第一版代码
3. 用户审查：标注问题（幻觉/安全/性能）
4. AI 修复，用户再审查
5. 循环直到通过

学习映射：
- VIBE-007 AI 代码评审
- VIBE-009 AI 辅助调试
- VIBE-010 任务拆解
```

#### 模式 C：Bug 修复挑战（Bug Hunt Challenge）

```
用户流程：
1. 系统给出含 Bug 的代码 + 错误现象
2. 用户限时定位 Bug
3. 修复并提交
4. 系统验证修复 + AI 评分
5. 排位榜

学习映射：
- JS-013 错误处理与调试
- VIBE-009 AI 辅助调试
- REACT-012 错误边界
```

#### 模式 D：代码评审训练（Code Review Training）

```
用户流程：
1. 系统给出 AI 生成的代码（含预设问题）
2. 学生逐一标注问题
3. 提交评审报告
4. 系统对比标准评审清单
5. 给出评审准确率评分

学习映射：
- VIBE-007 AI 代码评审
- VIBE-015 AI 代码质量管控
```
